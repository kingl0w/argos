## Question

Should the v1.0 independence criterion (file-set disjointness + `depends_on:` exclusion, exact wording in ARCHITECTURE.md §Independence detection lines 104–109) be relaxed to permit parallel dispatch on a known set of "shared but cleanly merge-able" files? Concretely: should `argos independence` and the orchestrator (ARG1-022) treat a pair of tickets that share `argos/cli/__main__.py` (or other registration-style files) as **independent**, given that the project's actual merge experience to date has been that such pairs merge cleanly under the "keep-both-registrations" pattern?

This is not a question I can answer at the ARG1-021 level — the criterion is fixed in three canonical sources (ARG1-021 §Intent, ARCHITECTURE.md §Independence detection, orchestrator agent doc §Parallel dispatch behavior) and any relaxation is an ARCHITECTURE.md change requiring an ADR. ARG1-021 ships the strict criterion. The question is whether to file the follow-up ADR now (so ARG1-022 can consume the relaxed criterion at dispatch time) or leave the strict criterion in place for v1.0 and accept the throughput floor.

## Context

**The strict criterion (canonical, three sources):**

1. ARG1-021 §Intent (verbatim): *"Two tickets are independent iff their `files_touched:` sets are disjoint AND neither lists the other in `depends_on:` frontmatter."*
2. ARCHITECTURE.md §Independence detection line 106 (verbatim): *"Two tickets are independent iff their file sets are disjoint AND neither lists the other in `depends_on:` frontmatter."*
3. `argos/specs/v1.0/agents/orchestrator.md` §Parallel dispatch behavior lines 89–94 (verbatim): *"Their declared `files_touched:` sets … are pairwise disjoint. Both conditions must hold. If either is unknown or ambiguous for a candidate pair, treat the pair as **dependent** and serialize."*

ARG1-021 has shipped a faithful implementation of this criterion at `argos/cli/orchestrator/independence.py`. All six acceptance criteria pass live.

**The empirical evidence that the strict criterion is too conservative:**

The recent ARG1-011 / ARG1-012 parallel batch shipped successfully despite both branches editing `argos/cli/__main__.py`. STATE.md `Done this cycle` entries (id `2026-04-30T17:42:38Z-ARG1-011-done` and `2026-04-30T17:40:38Z-ARG1-012-done`) document the merge succeeded with no conflict. Other Layer-2 tickets (ARG1-020, ARG1-031, ARG1-041) merged in parallel batches that all touched `argos/cli/__main__.py`; STATE.md entries for those tickets explicitly invoke the *"keep both registrations"* merge pattern.

If the strict criterion had been gating those batches, every one of them would have been falsely serialized. Throughput would have been ~3× lower than what actually shipped. PRD success criterion #4 (≥2× parallel speedup) is harder to hit under strict.

**Three identifiable shared-registration sites:**

1. `argos/cli/__main__.py` — subcommand tuple (`PUBLIC_SUBCOMMANDS` / `INTERNAL_SUBCOMMANDS`) + `_print_usage` text + dispatcher branches. Three localized edits per ticket. Auto-merges under git's default text driver because the additions are at distinct line ranges within the same logical region (tuple, help block, dispatcher block).
2. `argos/specs/v1.0/STATE.md` — handled out of band by ARG1-052's custom merge driver (`argos/scripts/state-merge-driver.sh`), not by git's default driver. Already not a problem for parallel dispatch because the merge driver dedupes append-only blocks.
3. `.claude/agents/<name>.md` ↔ `argos/specs/v1.0/agents/<name>.md` mirror pair — no concurrent edits to date, but structurally similar (parallel sessions writing distinct agent definitions both touch the mirror dir). Would auto-merge under the default driver if the per-file content is in distinct files; would conflict if two sessions edit the same agent file.

**The strict criterion is conservative-correct but tight:**

Per ARCHITECTURE.md §Invariants line 274 (verbatim): *"if independence analysis fails or is unavailable, the orchestrator falls back to serial dispatch (degraded but correct)."* The strict criterion never produces a wrong dispatch. It only produces serialization where parallelism would in fact have worked. ARG1-021's coder believes the right move at this layer is to ship the strict criterion as written and surface this question to the operator for a separate decision.

## Options considered

### A — Hard-coded carve-out allowlist

Add a config key `orchestrator.independence_carveout_files = ["argos/cli/__main__.py", ".claude/agents/", "argos/specs/v1.0/agents/"]` (or similar). The independence detector treats files in the allowlist as if they were not in `files_touched`. Document the allowlist criterion: *"a file qualifies for the allowlist iff its concurrent edits resolve cleanly under the configured merge strategy across all observed historical cases."*

**Pros:**
- Closes a measured throughput gap immediately. The three observed sites cover ~all parallel batches to date.
- Static — no merge-time inference, no risk of "looked safe at planning time, conflicted at merge time."
- Reversible. If a carve-out file later gains semantic-coupling content (not just registrations), drop it from the list.

**Cons:**
- The allowlist is a load-bearing project constant. Future contributors will trip over "why does adding X to file Y break dispatch?" questions; the allowlist needs a documented rationale per entry.
- Doesn't generalize. New shared-registration sites need explicit additions; the rule is "edit the allowlist when you add a new such site," which is brittle.
- Conflicts with the spirit of ARCHITECTURE.md §Independence detection (which is the orchestrator's only parallelism guarantee — file-set disjointness with no exceptions).

### B — Merge-strategy-aware criterion (principled relaxation)

Generalize the criterion: *"Two tickets are independent iff their files_touched sets are disjoint **on files where the configured merge strategy is `text` (the default driver)**, OR the shared file is registered with a custom merge driver in `.gitattributes` that is known to dedupe appends (e.g. STATE.md), OR the shared file matches a registration-style heuristic (e.g. concurrent edits land at distinct line ranges)."*

The simplest static approximation: read `.gitattributes` and treat any file whose merge driver is registered (e.g. STATE.md → `argos-state` driver) as a non-conflict; everything else uses the strict file-disjointness rule.

**Pros:**
- Principled — the criterion now corresponds to a property of the merge configuration, not an ad-hoc list.
- Self-extending: when ARG1-052's merge driver is extended to cover a new file, that file automatically becomes non-conflicting for parallel dispatch.

**Cons:**
- The hard part — *static prediction of clean merge for `argos/cli/__main__.py`* — is not solved by reading `.gitattributes`. `__main__.py` uses the default `text` driver; concurrent edits there merge cleanly only because of the human-chosen registration pattern, not because git knows anything special about the file. To capture `__main__.py` you still need a heuristic ("registration-style edits at distinct line ranges") which is hard to compute from a `files_touched:` list alone — the planner doesn't surface line ranges, only file paths.
- Implementation cost is higher than the carve-out option, and the heuristic is itself fragile in a way the carve-out is not.

### C — Status quo (strict criterion, accept the throughput floor)

Leave ARG1-021's strict implementation in place for v1.0. Document the gap in STATE.md §Known drift. Revisit after dogfooding ARG1-022 surfaces actual throughput pain.

**Pros:**
- ARG1-021 is fully shipped per the canonical specs; no ADR amendment needed.
- "Degraded but correct" is the explicit invariant. We are not in a zone where being wrong at dispatch is acceptable.
- The throughput cost is real but bounded. PRD success criterion #4 (≥2× speedup) is achievable even under strict — most of the v1.0 ticket batches have file-disjoint scopes; only the registration-touching ones are affected.

**Cons:**
- Locks in a known-suboptimal default. ARG1-022 will be measurably slower than necessary on Layer-2-shaped ticket cohorts.
- Punts the question rather than answering it. If we ship v1.0 to other users in this state, we ship a known throughput tax.

## Why escalated

This is exactly the shape of the four prior load-bearing escalations on this project (ADR-001 language, tomllib-vs-tomli, `.gitignore` directory precedence, ADR-002 AC harness portability). Specifically:

1. **The decision is in three canonical specs (ARCHITECTURE.md, ticket, agent doc) all aligned on the strict criterion.** Anything else is an ADR amendment, not a coder judgment call.
2. **Multiple tickets are affected.** ARG1-022 is the immediate consumer, but the relaxation question also affects every future Layer-2 cohort and the throughput floor of the orchestrator overall.
3. **The empirical evidence is real but the policy question is genuine.** "ARG1-011/012 merged cleanly" is a fact, but "should the criterion be relaxed because of that fact" is an operator decision about how much risk to accept at dispatch time vs. how much throughput to leave on the table.
4. **The four prior escalations were all this shape.** Each made an implicit assumption explicit; each landed an ADR; none was a "the agent should have just decided" call. This one matches.

## Recommendation

**Option C (status quo) for ARG1-021 itself**, because relaxing the criterion is by definition out of scope for the ticket as written. ARG1-021 ships strict.

**Operator decision needed on whether to file a follow-up ADR/ticket** for either Option A (hard-coded carve-out) or Option B (merge-strategy-aware) so ARG1-022's dispatch loop can consume the relaxed criterion. If neither is filed, ARG1-022 inherits the strict criterion and the throughput floor is what it is.

If the operator picks Option A, the follow-up ticket is ~30 lines plus tests plus an ADR amendment to ARCHITECTURE.md §Independence detection adding the carve-out clause. If the operator picks Option B, scope is materially larger (heuristic design + tests + ADR amendment + likely a new schema for `files_touched:` to surface line ranges).

This escalation is **advisory**, not blocking — ARG1-021 is fully shipped against the canonical specs and ARG1-022 can dispatch against the strict criterion. The escalation captures the gap so the operator can decide whether to close it before ARG1-022 lands.
