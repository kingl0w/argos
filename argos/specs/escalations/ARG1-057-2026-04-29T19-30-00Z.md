## Question

Should AC verification commands across v1.0 tickets (frontmatter validation, structured-block validation) use (a) a stdlib-only YAML-subset parser exposed as a new internal CLI subcommand, (b) `pyyaml` permitted as a dev-only dependency under ADR-001's existing `[project.optional-dependencies] dev` allowance, (c) a broader dev-deps allowlist, or (d) interpreter-pinning the AC harness to ensure pyyaml availability? This needs to resolve before Layer 2 fans out, because three Layer 2 tickets (020, 031, 041) will copy the pattern set by ARG1-010 / ARG1-012.

## Context

**ADR-001 §Decision item 2** (verbatim): *"Stdlib-only at runtime. Zero entries in `[project.dependencies]` in `pyproject.toml`. Any future need for a third-party runtime dependency requires an ADR. Test-only dev dependencies (`pytest`) live in `[project.optional-dependencies] dev` and are not part of the install critical path."*

**ADR-001 §Consequences/Test framework** (verbatim): *"Tests use `unittest` from the stdlib (precedent: ARG1-030, ARG1-050). `pytest` is allowed as an optional dev dependency for ergonomics but is not required to run the AC verification."*

**ARG1-010 AC#3** (the trigger): `python -c "import yaml,sys; d=yaml.safe_load(open('.claude/agents/orchestrator.md').read().split('---')[1]); sys.exit(0 if 'allowed_tools' in d and 'denied_paths' in d else 1)" exits 0`. `pyyaml` is third-party; not in `pyproject.toml`.

**ARG1-010 verifier disposition.** AC#3 passed locally only because the AC was run under `/usr/bin/python3` (which had a system-installed pyyaml 6.0.1). The repo's project venv at `/home/user/.graphify-venv/bin/python3` does NOT have pyyaml — the same AC command exits with `ModuleNotFoundError: No module named 'yaml'`. The harness reached for the system interpreter by chance, not by design.

**ARG1-030 chose differently.** ARG1-030's plan §Risks line 120 (verbatim): *"No yaml dep. Hand-rolled parser is fragile if the structured-block grammar grows. Acceptable for v1.0 because the grammar is deliberately tiny (six keys total). Future tickets can swap in `pyyaml` via an explicit plan step."* That explicit plan step never happened — ARG1-010 just used pyyaml without filing one.

**Other affected tickets.** `grep` across `argos/specs/v1.0/tickets/` finds the same `python -c "import yaml..."` pattern in:
- `argos/specs/v1.0/tickets/ARG1-010-orchestrator-agent-definition.md:27` (shipped — AC#3)
- `argos/specs/v1.0/tickets/ARG1-012-dispatch-log-writer.md:25` (queued — AC#1, dispatch-log frontmatter)

ARG1-020 (worktree spawn) and ARG1-041 (escalation writer) do not currently have yaml ACs, but the user has named them as Layer 2 tickets that will likely copy ARG1-010's pattern.

**Shipped python is stdlib-only.** No `import yaml` in `argos/cli/` (audited via `grep -rn`). `pyproject.toml` has `dependencies = []` and `dev = ["pytest>=7"]`. The reference parsers (`state_parser.py`, `verifier_parser.py`, `escalation_validator.py`) all hand-roll their YAML-subset parsing. ARG1-040's plan line 104: *"(No PyYAML — flat scalar key:value only.)"*. Project evidence consistently rejects pyyaml.

**Structural shape of the parsing target.** Subagent frontmatter has flat scalars (`name`, `description`) plus block sequences (`allowed_tools:` + indented `- Read` items, `denied_paths:` similar with one quoted value). The existing `escalation_validator.py` parser handles flat scalars only; extending to block sequences is ~30–50 lines plus a new internal subcommand surface (e.g. `argos frontmatter-keys <file> --require K1,K2`).

## Options considered

### A — Stdlib YAML-subset parser, exposed as a new internal CLI subcommand

ACs are rewritten from `python -c "import yaml..."` to `argos frontmatter-keys <file> --require allowed_tools,denied_paths` (exit 0 iff all required keys present). New module under `argos/cli/frontmatter_parser.py`, parallel in shape to `state_parser.py` / `verifier_parser.py` / `escalation_validator.py`. Handles: flat scalars, block sequences, double-quoted scalar values (the only YAML feature the orchestrator frontmatter uses beyond flat scalars). Out-of-scope: anchors, multi-line strings, nested mappings — fail loudly if encountered, since none of the actual frontmatter we control needs them.

**Pros.**
- Fully consistent with ADR-001's stdlib-only mandate, with no need to amend it.
- Parallels every existing reference parser in the project — sets the same pattern Layer 2 tickets are already inheriting (ARG1-040's hand-rolled approach is already shipping in `escalation_validator.py`).
- AC verification becomes interpreter-independent: any python3 ≥3.9 satisfies it, no `pip install` step required, no system-package dependency.
- Reusable: ARG1-012, ARG1-020, ARG1-031, ARG1-041 all get the same `argos frontmatter-keys` tool. One implementation, many ACs.
- Closes the existing inconsistency where ARG1-030's plan promised "future tickets can swap in pyyaml via an explicit plan step" but ARG1-010 silently bypassed that gate.

**Cons.**
- ARG1-010 already shipped with the pyyaml AC. Forward fix means a follow-up ticket to retrofit ARG1-010's AC#3 to invoke the new tool, not a retro-edit (per this ticket's "do not modify ARG1-010's shipped files" constraint).
- ~30–50 lines of YAML-subset code, plus its own tests. Modest, but not free.
- Hand-rolled parsers accumulate edge cases. The grammar must be pinned narrowly and documented.

### B — Permit `pyyaml` as a dev-only dependency under ADR-001's existing dev-extras mechanism

Amend `pyproject.toml` `[project.optional-dependencies]` to add `pyyaml>=6` to the `dev` extra. ACs continue to use the inline `python -c "import yaml..."` pattern, but the tooling section of CONTRIBUTING / README documents `pip install -e .[dev]` as a prerequisite. ADR-001 §Decision item 2 already grants the dev-extras escape hatch (pytest precedent); this is the same mechanism applied a second time.

**Pros.**
- No new code. Drop-in: ARG1-010's existing AC#3 starts passing under the project venv after `pip install -e .[dev]`.
- pyyaml is the de facto industry-standard YAML parser; its semantics are unsurprising and battle-tested.
- Honest about tooling: makes explicit what ARG1-010 implicitly required.

**Cons.**
- Conflicts with the strong project pattern of hand-rolling YAML subsets (state_parser, verifier_parser, escalation_validator all explicitly reject pyyaml). Adds a dep where stdlib suffices.
- Couples AC verification to a third-party package even though the runtime explicitly does not. Two boundaries (runtime vs AC-verification) that the project keeps having to re-explain.
- Surface area grows: every ticket reviewer must remember "ACs may use pyyaml but runtime mustn't." Easy to drift into runtime use.
- Doesn't help Cursor / Codex / Gemini harness mirrors that may not run `pip install -e .[dev]` before AC checks.

### C — Broader dev-deps allowlist (e.g. pytest, pyyaml, plus future deps as needed)

Amend ADR-001 to enumerate a small explicit allowlist of dev dependencies and the criteria for adding to it. Functionally a superset of (B).

**Pros.**
- Future-proofs against the next "we need this one library for tooling" debate.

**Cons.**
- Premature. We have exactly one motivating dep (pyyaml) and one motivating use case (AC frontmatter validation). Generalizing now is YAGNI; the same outcome is achievable by filing a small ADR each time a new dev dep is needed.
- Erodes ADR-001's clean "stdlib-only at runtime, pytest as the named test exception" rule into a list that grows over time.

### D — Pin the AC harness to a specific interpreter (e.g. `/usr/bin/python3`)

Document that ACs must be run under the system interpreter where pyyaml is reliably installed. CI uses an explicit `python3` from a known image.

**Pros.**
- Zero code change.

**Cons.**
- Brittle. Doesn't fix the inconsistency, just hides it behind one specific harness configuration. Solo developers on machines without system pyyaml hit the same wall.
- Conflicts with `argos init`'s portability ambition (PRD Distribution: argos installer ships to "every dev machine that has python3"). Pinning to system python3 specifically excludes pyenv-based and venv-based contributor setups.
- Doesn't solve the dogfooding constraint: this repo's own venv is the de facto contributor setup, and it doesn't have pyyaml.

## Why escalated

This is a **load-bearing technology choice with an existing ADR in conflict**, exactly matching ARG1-010's escalation trigger #2 ("Stdlib / third-party boundary ambiguity"). The fix-in-place gate from the ticket prompt (*"unambiguously option (a) AND no other ticket needs pyyaml AND the stdlib path is a few lines"*) is not unambiguously met:

1. **Multiple tickets are affected, not one.** ARG1-010 (shipped), ARG1-012 (queued), and at least three Layer 2 tickets (020, 031, 041) per the user's named scope. The pattern set here will be copied — fixing it in one ticket does not generalize without an explicit decision recorded somewhere ADR-shaped.

2. **The stdlib path is not a few lines.** It needs a new internal subcommand (parallel to the four existing ones) and a YAML-subset parser handling block sequences. ~30–50 lines of code plus tests plus dispatch wiring plus a follow-up ticket to retrofit ARG1-010's existing AC. That is small in absolute terms but is "a small new tool surface," not "a few lines."

3. **The choice between (A) and (B) is genuinely contested by the project's own evidence.** ADR-001 explicitly opens the dev-extras escape hatch (pytest precedent), which makes (B) defensible. Every existing reference parser explicitly closes that hatch by hand-rolling stdlib YAML subsets, which makes (A) the established pattern. Either reading is valid; the project hasn't adjudicated which boundary applies to AC verification specifically.

4. **Calibration with prior escalations.** ADR-001 itself was filed because three tickets had merged on the same kind of de-facto-Python assumption; the escalation made the implicit explicit. The tomllib-vs-tomli call was filed for an exactly-shaped stdlib-vs-third-party question. The `.gitignore` directory-precedence call was filed for a load-bearing semantics question that affected multiple tickets. This escalation matches all three priors in shape: an existing ADR is silent or ambiguous on a specific boundary; the choice will be copied across tickets; the wrong default is hard to undo.

## Recommendation

**Option (A).** Reasons:

- It is fully consistent with ADR-001 as written. No amendment needed.
- It matches the established project pattern: every existing reference parser is stdlib-only, and ARG1-040 explicitly chose the hand-rolled path. Option (B) would be the inconsistent choice given the project's actual evidence.
- The reusability gain is real. One `argos frontmatter-keys` tool serves ARG1-010's AC#3, ARG1-012's AC#1, and the three Layer 2 ticket ACs that don't exist yet. The marginal cost per future AC is one shell line, not "ensure pyyaml is in the right environment."
- It keeps AC verification interpreter-agnostic, which matters for the multi-harness ambition (Cursor / Codex / Gemini may not bootstrap `[dev]` extras before running AC checks).
- The forward-fix ticket to retrofit ARG1-010's AC#3 is small (rewrite one AC line, no shipped-file changes outside the test surface). The retrofit becomes a clean Layer 2 prerequisite, not a regression.

If you accept (A): the next ticket spec should be ARG1-NNN-frontmatter-keys-tool, defining the `argos frontmatter-keys <file> --require K1,K2[,...]` subcommand, then a follow-up ARG1-NNN to retrofit ARG1-010's and ARG1-012's ACs in one batch (so the pattern is established before Layer 2 fans out).

If you reject (A) and prefer (B): file a one-paragraph amendment to ADR-001 §Decision noting that pyyaml is permitted in `[project.optional-dependencies] dev` for AC frontmatter validation specifically, with a "no runtime use" guard. Then the existing AC pattern stands as-is, and the only follow-up is updating `pyproject.toml` and CONTRIBUTING.

Either way, the decision needs to be recorded as an amendment to ADR-001 or as a new ADR — not just as ticket prose — so future contributors have a single canonical reference. The current state (ARG1-030 plan saying one thing, ARG1-010 doing another, ADR-001 ambiguous on the AC-vs-runtime distinction) is the drift this escalation should close.

## Resolution

**Drained:** 2026-04-29 by operator (ianfrushon@gmail.com).

**Decision:** Option (A) — stdlib-only YAML-subset parser exposed as a new internal CLI subcommand `argos frontmatter-parse`.

**Operator reasoning (verbatim from drain session):** v1.0 is intended for distribution to other users via the (still-deferred) packaging channel from ARG1-001. That distribution property requires AC tooling to work on any machine where `argos` works, without dev-extras setup. (B) would require every downstream user to install pyyaml separately and would silently pass ACs against system pyyaml when they didn't — the same foot-gun that surfaced this escalation. (A) makes the parser part of the tool.

**Authoritative record:** [`argos/specs/v1.0/decisions/ADR-002-ac-harness-portability.md`](../v1.0/decisions/ADR-002-ac-harness-portability.md). ADR-002 pins the subset grammar (block sequences of strings, flat scalars at depth 1, no flow style, no anchors/aliases/tags, no multiline scalars, no nested mappings beyond depth 1, comments preserved as no-ops, UTF-8 only) and the rejection contract (any input outside the subset exits non-zero with line number and reason). Frontmatter drifts to stay inside the parser, not vice versa.

**Follow-ups filed:**
- `ARG1-060` — implement the `argos frontmatter-parse` subcommand (`argos/cli/frontmatter_parser.py`, registration in `argos/cli/__main__.py`, tests). depends_on ADR-002.
- `ARG1-059` — retrofit existing AC text to invoke the new subcommand instead of `import yaml`. depends_on ARG1-060. AC-text-only; no shipped-file changes to ARG1-010 / ARG1-012 outputs.

**Layer 2 gating:** Layer 2 fan-out (ARG1-020 / ARG1-031 / ARG1-041) blocks on ARG1-060 landing. ARG1-059 should land in the same merge window as ARG1-060 so the pattern is established before any Layer 2 ticket can copy the old `import yaml` shape.
