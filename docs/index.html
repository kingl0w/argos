argos builds, the operator owns the merge.

Queue a ticket. Argos plans it, writes the code and tests in an isolated git worktree, verifies against acceptance criteria, and pushes a branch. It never merges — you do. Spec-driven outer-loop orchestration for AI coding agents: a scaffold for Claude Code, Cursor, Codex CLI, and Gemini CLI, plus a stdlib-only Python CLI that runs the whole loop headlessly.

Run it on your repo

Argos works on any codebase — your repo declares its own language, dependencies, and test command in argos/conventions.md, and every dispatched session builds to those rules.

git clone https://github.com/kingl0w/argos.git
cd your-repo
argos init            # scaffold argos/specs/, write conventions.md, install hooks
$EDITOR argos/conventions.md   # your language, deps, test command
argos queue add ABC-001        # queue a ticket you wrote under argos/specs/
argos orchestrate     # builds, tests, verifies, pushes a branch — you merge
Why it's built this way

  1. Specs are the source of truth

    Markdown specs — PRD, architecture, tickets, ADRs, STATE.md — drive every session. Hooks keep them honest: STATE.md is append-only and argos status fails when the specs contradict themselves. The live graph above is projected straight from them.

  2. Five agents, each with a narrow job

    The planner never writes code. The coder can't touch specs. The watchdog diffs code against plan and halts on drift. Only the verifier runs tests and updates state. Narrow tool permissions make the loop auditable instead of hopeful.

  3. The human owns the merge

    Sessions run in parallel worktrees when independence detection clears them, and every one ends the same way: a pushed branch, never a silent merge. Ambiguity escalates to the operator — there is no silent recovery.
